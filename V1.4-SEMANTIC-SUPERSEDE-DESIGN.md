# v1.4 Semantic Supersede Design

## 目标

从 rule-based conflict resolution 演进到 rule-guided semantic supersede，提升 memory 质量和长期 recall 准确性。

## 当前问题（v1.3）

### Rule-based 的局限

```typescript
// 当前逻辑：只能处理明确定义的冲突
if (memoryGroup === "preference:language" && topics.includes("chinese")) {
  supersede(oldMemory);
}
```

**问题场景：**

1. **语义相近但表述不同的更新无法识别**
   - 旧：`"User prefers concise Chinese replies."`
   - 新：`"用户希望简洁的中文回复。"`
   - 当前：两条都保留（重复）
   - 应该：识别为同一偏好，supersede

2. **部分更新无法合并**
   - 旧：`"Project uses TypeScript and Node.js 18."`
   - 新：`"Project upgraded to Node.js 24."`
   - 当前：两条都保留或全部替换
   - 应该：合并为 `"Project uses TypeScript and Node.js 24."`

3. **规则维护成本高**
   - 每个新的 memoryGroup 需要手写规则
   - 边界情况越来越多
   - 规则之间可能冲突

### 量化当前问题

基于 v1.3 的实际使用数据（需要通过 benchmark 验证）：

| 指标 | v1.3 当前值 | 问题 |
|------|------------|------|
| 重复 preference 率 | ~15-20% | 语义相同但表述不同的偏好被重复存储 |
| 过时 semantic 残留 | ~10-15% | 项目事实更新后，旧版本未被清理 |
| 无效 recall 率 | ~8-12% | 检索到的 memory 中有部分已过时或重复 |
| 规则覆盖率 | ~60-70% | 只有明确定义的 memoryGroup 有冲突处理 |

## v1.4 方案：Rule-Guided Semantic Supersede

### 核心思路

**不是替换 rule-based，而是增强它：**

```
rule-based (v1.3)
    ↓
rule-guided semantic supersede (v1.4)
    ↓
    = rules 定义边界 + semantic similarity 识别冲突
```

### 三层判断机制

```typescript
function shouldSupersede(existing: Memory, candidate: Memory): SupersedeDecision {
  // Layer 1: Rule gate (快速过滤)
  if (!isSameKind(existing, candidate)) return "coexist";
  if (!isSameScope(existing, candidate)) return "coexist";

  // Layer 2: Semantic similarity (识别潜在冲突)
  const similarity = cosineSimilarity(existing.embedding, candidate.embedding);
  if (similarity < 0.75) return "coexist"; // 不够相似，共存

  // Layer 3: Conflict detection (确认冲突类型)
  if (similarity >= 0.92) {
    // 高度相似 = 重复或重述
    return candidate.createdAt > existing.createdAt ? "supersede" : "skip";
  }

  if (similarity >= 0.75 && hasContradiction(existing, candidate)) {
    // 中度相似 + 有矛盾词 = 更新
    return candidate.createdAt > existing.createdAt ? "supersede" : "coexist";
  }

  return "coexist";
}
```

### 配置开关

```typescript
// config schema 新增
memory: {
  conflictResolution: "rule-based" | "semantic" | "hybrid", // 默认 "rule-based"
  semanticSimilarityThreshold: 0.75, // 相似度阈值
  supersedeHighSimilarity: 0.92,     // 高相似度阈值（视为重复）
}
```

## 风险和缓解

### 风险 1: 误 supersede 导致信息丢失

**缓解：**
- 默认保持 `rule-based` 模式
- 提供 audit trail 和 rollback
- 相似度阈值保守设置（0.75）

### 风险 2: 性能影响

**缓解：**
- 只在同 kind + 同 scope 内计算相似度
- 使用已有的 embedding（不额外计算）
- 批量操作时可选禁用 semantic check

### 风险 3: 用户困惑

**缓解：**
- 清晰的配置文档
- `doctor` 命令显示当前模式
- `memory explain` 显示 supersede 原因

## 附录：矛盾词检测

```typescript
const CONTRADICTION_PATTERNS = {
  version: /(\d+\.\d+)/g,  // "Node 18" vs "Node 24"
  boolean: /(enable|disable|true|false|yes|no)/gi,
  preference: /(prefer|like|want|avoid|dislike)/gi,
  negation: /(not|no|never|don't|doesn't)/gi,
};

function hasContradiction(a: Memory, b: Memory): boolean {
  // 检测版本号冲突
  const aVersions = a.content.match(CONTRADICTION_PATTERNS.version);
  const bVersions = b.content.match(CONTRADICTION_PATTERNS.version);
  if (aVersions && bVersions && aVersions[0] !== bVersions[0]) {
    return true;
  }

  // 检测布尔值冲突
  const aBool = extractBoolean(a.content);
  const bBool = extractBoolean(b.content);
  if (aBool !== null && bBool !== null && aBool !== bBool) {
    return true;
  }

  // 检测否定词
  const aNeg = CONTRADICTION_PATTERNS.negation.test(a.content);
  const bNeg = CONTRADICTION_PATTERNS.negation.test(b.content);
  if (aNeg !== bNeg) {
    return true;
  }

  return false;
}
```

1. **可验证**：每个改进指标都有对应的 benchmark 测试

3. **可回滚**：audit trail 确保不会丢失重要信息
4. **渐进式**：不是推倒重来，而是在现有基础上增强

**关键数字（需要测试验证）：**
- 重复率：15-20% → 5-8%（减少 60-70%）

- 无效 recall：8-12% → 3-5%（减少 50-65%）
